Introduction

In this assessment, you are required to develop a partial compiler for the language PAL (Pretty Awful Language). You will only be building the frontend of the compiler: you need to implement parsing and semantic analysis, but you don't need to implement artifact generation.

Please read through the whole of this document carefully before starting work, including the Submission and Grading Criteria sections.

If you have any questions, please contact me (a.sampson@abertay.ac.uk).

Software Requirements

The overall objective of this assignment is to develop a parser and semantic analyser for the language PAL, encapsulated within a command-line application that will enable a user to process a PAL program to verify its syntactic and semantic correctness.

Your primary source document for this is the EBNF specification of PAL, which includes commentary on the semantics of the language.

I will run your compiler application inside an automated test harness as part of the marking process, so you need to keep the compiler's interface very simple:

  • Your compiler application must take the name of the PAL program file as its only command-line argument (i.e. in C#, the first element of the array of arguments passed to Main).
  • Error messages should be printed to standard output, with the compiler printing absolutely nothing if the PAL program is correct.
  • The compiler must not expect any user input other than the command-line arguments, or display a GUI.

I strongly suggest that you use C# and the Ardkit framework — which we used in previous practicals — to implement your compiler. If you'd like to use a different language or framework, I'm happy for you to do so, but you must contact me by email to confirm that it'll meet the assessment requirements before starting work.

I've provided a pre-written scanner for PAL for you to use as the basis for your compiler; see the PALScanner source file (PALScanner.cs) and (for reference) the FSM that it implements (PAL.FSM.txt). This source file is complete; please don't modify it in any way.

Suggested Activities

Following is a methodology that you can follow in developing your compiler. This process will enable you to build up your application step by step, testing each stage as it is completed so that you know the next step is being built on a sound foundation:

  1. Review the EBNF specification for PAL (PAL.EBNF.txt), noting the different forms that the declarations and statements can take.
  2. Write some programs in the language, to check your understanding of the syntax and to act as test programs for the compiler as it is developed.
  3. Download the C# scanner file that has been provided.
  4. Ensure you know how to use the Ardkit framework classes by referring back to previous lectures, practicals and examples; ask if you are in doubt.
  5. Develop a "pure" recursive-descent parser for PAL. This will be defined as a class inheriting from the RecoveringRdParser class in the Ardkit compiler toolkit library.
  6. Write the main application that will instantiate the parser and provide an appropriate command-line interface.
  7. Test your "pure" parser by running your application against a suite of test programs. Your testing should ensure that valid programs are verified as such, and that syntactically invalid programs have their errors reported correctly.
  8. Add semantic analysis to the parser. This will involve developing a semantic analysis class and symbol table mechanism, based around the Ardkit compiler toolkit framework. Ardkit includes a range of error classes that should be sufficient for most projects; you can also add new error classes if you feel you need them.
  9. Test that semantic analysis works correctly by again running your application against your test suite. This should include valid programs and programs that are syntactically valid but semantically invalid.

You should now have a complete and correct parser and semantic analyser for the PAL language.

Test suite

As described above, you need to come up with a suite of PAL test programs to ensure that your program handles both valid and invalid programs correctly. Please refer back to advice in previous modules about how to design test suites — for example, thinking about coverage of the specification and/or your compiler code, and about testing unusual cases in the code.

I encourage you to exchange test programs with other students. Since you're all implementing the same language specification, this is a good way of finding problems in your implementation that you might otherwise not have thought of. It's all right to include test programs that came from other students along with your own tests in the submission, but these must be clearly distinguished from your own work (e.g. by putting them in a different folder) — and you shouldn't be relying entirely on other students' tests!

Grading Criteria

Unit 1 will be assessed using the following criteria;

  • the correctness and completeness of your application when executed against my test suite of valid and invalid PAL programs, and
  • the quality of your code and test suite.

The maximum grade that can be achieved depends on the extent to which you have completed the compiler implementation in terms of both parsing and semantic analysis.

Maximum Grade Description
A The application fully and correctly implements parsing and semantic analysis.
B Parsing and semantic analysis are fully implemented but with some minor errors in either or both.
C Parsing is implemented (perhaps with some minor errors) but semantic analysis has major omissions or errors.
D Parsing is implemented (perhaps with some major omissions or errors) but with little or no semantic analysis.
MF The submission could meet the minimum D standard with some more work under minimal direction.
F Substantial work and direction would be required to bring the submission up to the minimum D standard.
NS No submission, or the submission contained no relevant material.

The above maximum grades will be awarded if the quality of the code and test suite reflects the good practice taught in previous modules — in particular, your code and tests should be written so that they are easy to read, following consistent naming and formatting conventions, and could be maintained by a future developer. Grades below the maximum will be awarded for code that does not follow good development practice.
